# 宇树 G1

本文档说明 G1 在本工作区中有哪些控制入口、各自负责什么，以及它们对应的官方 ROS 2 话题。先按需求选择控制路径，再查后面的消息和 API 细节。

| 需求 | 使用入口 | 责任边界 |
|---|---|---|
| 行走、站立和内置动作 | `/api/sport/*`、`/api/arm/*` | Unitree 固件负责轨迹、平衡和电机控制 |
| Unitree 行走 + 自定义上肢 | `/api/sport/*` + `/arm_sdk` | 固件控制下肢，用户发布腰部和手臂目标 |
| 本工作区整机位置控制 | FPC 或 JTC | controller 生成 31 轴位置，`G1TopicSystem` 补齐 MIT 字段并发布 `/lowcmd` 和夹爪命令 |
| 完全自定义底层控制 | 直接 `/lowcmd` | 用户负责全部本体关节、安全、模式、MIT 参数和 CRC；不能保留内置行走 |

## 本工作区控制结构

机器人本体使用官方 [`unitree_ros2`](https://github.com/unitreerobotics/unitree_ros2) 消息定义。G1 只需要以下两个包：

- `unitree_api`：机器人服务请求/响应消息。
- `unitree_hg`：G1/H1 系列的状态与控制消息。

本项目不编译 `unitree_go` 和 `unitree_ros2_example`。

整机控制使用 ROS 2 Foxy 原生 `ros2_control`：

- `unitree_g1_description` 只保存整机模型、mesh 和硬件无关的 ros2_control 资源声明；
- `unitree_g1_ros2_control` 提供真实 `SystemInterface`、C++ forward-position controller、标准 JointTrajectoryController 实例和状态 broadcaster；
- `robot_bringup` 只组合设备与控制栈，`whole_body_dashboard.launch.py` 只连接已有 `/controller_manager`；
- 同一份展开 URDF 同时交给 `controller_manager` 与 `robot_state_publisher`，不存在第二套 controller-manager facade。

统一硬件插件导出 31 个位置命令接口（G1 29 轴和左右 Gloria-M 偏心轴），另有 93 个关节状态接口、双 KWR57 六轴力状态以及 pelvis IMU。`forward_position_controller`（FPC）与 `joint_trajectory_controller`（JTC）请求完全相同的 31 个接口，由 controller_manager 保证只能激活其中一个。claim 只表示命令所有权，不自带反馈时效语义；插件分别以 `0.25 s` 检查 G1、以 `0.75 s` 检查 Gloria。夹爪反馈过期时只跳过对应 MIT 输出，反馈恢复后自然恢复，不中断本体 LowCmd。manager 以 500 Hz 调用硬件 `read()/write()`，KWR57 原始 1 kHz 话题保持不变。


## G1 官方话题、控制方法与模式
本节只整理 `src/unitree_ros2` 当前源码中可验证的 G1 接口，并补充与该版本相关的官方固件接口。

表中的名称是 `ros2 topic list` 所见的 ROS 2 名称。官方 DDS 文档常写 `rt/lowcmd`、`rt/arm_sdk`；`rt/` 是 ROS 2 到 DDS 的底层话题前缀，在本仓库的 ROS 2 节点中应使用 `/lowcmd`、`/arm_sdk`，不要改成 `/rt/lowcmd`。

### G1 话题速查

#### 本体、上肢与高层服务

| ROS 2 话题 | 消息类型 | 方向 | 用途与当前源码依据 |
|---|---|---|---|
| `/lowstate` | `unitree_hg/msg/LowState` | G1 -> PC2 | 高频整机状态：`mode_pr`、`mode_machine`、IMU、35 个消息槽位的 `motor_state`、遥控器字节和 CRC。G1 示例使用前 29 个本体电机状态，并注明高频流为 500 Hz。 |
| `/lf/lowstate` | `unitree_hg/msg/LowState` | G1 -> PC2 | `/lowstate` 的低频版本；具体频率由固件决定。`g1_low_level_example` 通过编译期 `HIGH_FREQ` 在二者间选择。 |
| `/lowcmd` | `unitree_hg/msg/LowCmd` | PC2 -> G1 | 全身低层电机命令，包含 `q/dq/tau/kp/kd`、`mode_pr`、`mode_machine` 和 CRC。它接管 G1 本体控制，不能与内置行走控制或另一个 `/lowcmd` 发布者并行使用。 |
| `/secondary_imu` | `unitree_hg/msg/IMUState` | G1 -> PC2 | 躯干第二 IMU；`g1_ankle_swing_example` 单独订阅该话题。主 IMU 同时包含在 `/lowstate.imu_state` 中。 |
| `/arm_sdk` | `unitree_hg/msg/LowCmd` | PC2 -> G1 | 高层运动服务下的腰部/上肢混合接管。有效命令槽位为 12–28，`motor_cmd[29].q` 是 $[0,1]$ 接管权重；腿部仍由内置移动控制器控制。 |
| `/api/sport/request` | `unitree_api/msg/Request` | PC2 -> G1 | `LocoClient` 高层运动请求：FSM、平衡模式、速度、摆腿/站立高度和内置上肢任务。 |
| `/api/sport/response` | `unitree_api/msg/Response` | G1 -> PC2 | 与 request 的 `identity.id` 配对，返回状态码和 JSON 数据。当前 `BaseClient` 的单次等待上限为 5 秒。 |
| `/api/arm/request` | `unitree_api/msg/Request` | PC2 -> G1 | `G1ArmActionClient` 的预定义上肢动作和动作列表请求。 |
| `/api/arm/response` | `unitree_api/msg/Response` | G1 -> PC2 | Arm Action 响应；API ID 与 sport 服务分别解释，即使数值相同也不是同一个调用。 |
| `/api/motion_switcher/request` | `unitree_api/msg/Request` | PC2 -> G1 | 查询、选择或释放当前运动服务所有权，也可设置 silent 状态。 |
| `/api/motion_switcher/response` | `unitree_api/msg/Response` | G1 -> PC2 | MotionSwitcher 响应。全身低层接管前应先确认内置运动服务已经释放。 |
| `/api/voice/request` | `unitree_api/msg/Request` | PC2 -> G1 | TTS、PCM 播放、音量和头部 RGB 灯控制；属于外设控制，不改变运动控制权。 |
| `/api/voice/response` | `unitree_api/msg/Response` | G1 -> PC2 | Voice API 响应。 |

`LowCmd` 和 `LowState` 的 ROS 消息固定有 35 个电机槽位，但 G1 23/29-DoF 本体只使用与硬件配置对应的前 29 个索引。槽位 29–34 是 Unitree 消息预留位，不是本项目两只 Gloria-M 的电机；Gloria-M 使用独立的 `gloria_ros` 消息和 CAN 驱动。只有在 `/arm_sdk` 协议中，索引 29 的 `q` 字段被重新定义为上肢接管权重。


#### 固件有但当前源码快照未封装的接口

| 官方 DDS 名称 | 对应 ROS 2 名称 | 状态 |
|---|---|---|
| `rt/sportmodestate` | `/sportmodestate` | G1 固件 1.5.1 以后可发布 `fsm_id`、`fsm_mode`、`task_id`、`task_time`。当前 `unitree_hg/msg` 目录没有 `SportModeState.msg`，因此不能直接用本快照生成的 ROS 类型订阅；需要更新匹配固件的 `unitree_hg` 或自行加入官方 IDL。不要使用仓库中基于 `unitree_go/SportModeState` 的通用示例代替 G1 类型。 |
| `rt/user_lowcmd` | `/user_lowcmd` | 新版官方 User Development Mode 的全身临时接管话题，需要新版 `SwitchToUserCtrl()`/`SwitchToInternalCtrl()` 客户端。当前 `g1_loco_client.hpp` 未实现这两个调用，也没有使用该话题的 G1 示例。 |

`unitree_hg` 还定义了 `BmsState`、`MainBoardState` 等消息类型，但当前 G1 示例和头文件没有把它们绑定到确定的话题名，因此不能仅凭消息名推断机器人一定发布同名 topic。现场应以 `ros2 topic list -t` 为准。

`/joint_states`、`/robot_description`、`/tf`、`/pelvis_imu_broadcaster/imu`、FPC/JTC 命令接口以及 KWR57/Gloria-M 话题，是本工作区在官方 G1 接口之上生成的 ROS 2 接口，不是 G1 固件原生 topic；它们在后续“启动入口”和各功能包 README 中说明。

### 控制方法与控制权

| 控制路径 | 入口 | 控制范围 | 保留内置行走 | 主要约束 |
|---|---|---|---|---|
| Loco 高层运动 | `/api/sport/*` | 整机模式、平面速度、步态参数、内置上肢任务 | 是 | 输入的是机身速度/模式目标，不是腿关节命令；最终模式切换由固件状态机决定。 |
| Arm Action | `/api/arm/*` | 固件内置的挥手、握手等动作 | 是 | 仅支持指定 FSM；与 `/arm_sdk` 互斥。动作 ID 应通过 `GetActionList()` 查询，不应按其他固件版本硬编码。 |
| Arm SDK 混合接管 | `/arm_sdk` | 腰部和双臂索引 12–28 | 是 | 索引 29 为权重；持续发布期间 Arm Action 会返回“arm_sdk occupied”。适合“自定义上半身 + Unitree 下半身行走”。 |
| MotionSwitcher | `/api/motion_switcher/*` | 运动服务所有权，不直接产生关节目标 | 取决于所选服务 | `CheckMode()`/`SelectMode()`/`ReleaseMode()` 管理当前运动服务；释放内置服务后，高层 Loco 将失效，直到重新选择运动服务。 |
| 全身低层控制 | `/lowcmd` + `/lowstate` | G1 所有可用本体电机 | 否 | 用户负责姿态、平衡和安全；必须先释放内置运动服务，正确设置模式与 CRC，并保证只有一个命令发布者。 |
| User Development Mode | 新版客户端 + `/user_lowcmd` | 可临时在内置控制与自定义全身控制间切换 | 否 | 只见于较新的官方接口，本源码快照不能直接调用；切入和切出动作都必须回到标准站姿。 |
| Dex3 | `/dex3/*` | 独立灵巧手 | 是 | 仅适用于安装了 Dex3 的机器人；不占 G1 前 29 个本体电机索引。 |
| Voice | `/api/voice/*` | 扬声器、音量和 RGB 灯 | 是 | 非运动控制。 |

需要“自己控制上半身、Unitree 控制下半身”时，正确组合是 `/api/sport/*` 加 `/arm_sdk`。不要使用 `/lowcmd`，也不要运行会调用 `MotionSwitchClient::ReleaseMode()` 的 `g1_dual_arm_example`；该示例虽然名为 dual arm，实际会先停掉内置运动服务，再进入全身低层控制。

### Loco 高层 API

`LocoClient` 使用 `/api/sport/request` 和 `/api/sport/response`。API ID 只在该 endpoint 内有意义。

| API ID | 方法 | 参数/结果 | 含义 |
|---:|---|---|---|
| 7001 | `GetFsmId(int&)` | 输出 `fsm_id` | 当前内置运动状态机 ID。 |
| 7002 | `GetFsmMode(int&)` | 输出 `fsm_mode` | 当前状态是静态还是动态；见后面的模式表。 |
| 7003 | `GetBalanceMode(int&)` | 输出 `balance_mode` | 平衡站立或持续踏步。 |
| 7004 | `GetSwingHeight(float&)` | 输出摆腿高度 | 固件会约束有效范围。 |
| 7005 | `GetStandHeight(float&)` | 输出站立高度 | 固件会约束有效范围。 |
| 7006 | `GetPhase(vector<float>&)` | 输出步态相位 | 已标记 deprecated。 |
| 7101 | `SetFsmId(int)` | `fsm_id` | 请求切换运动状态；调用成功只表示请求被接受，最终状态仍需再次查询。 |
| 7102 | `SetBalanceMode(int)` | `balance_mode` | 设置平衡/持续踏步策略。 |
| 7103 | `SetSwingHeight(float)` | 摆腿高度 | 设置步态参数。 |
| 7104 | `SetStandHeight(float)` | 站立高度 | 设置步态参数。 |
| 7105 | `SetVelocity(vx, vy, omega, duration)` | 前向、横向、偏航速度和有效秒数 | 机身平面速度设定值；固件生成腿部轨迹、平衡和力矩，并自动裁剪超范围目标。 |
| 7106 | `SetTaskId(int)` | 内置上肢任务 ID | 与 `/api/arm` 的 7106 是不同 endpoint 下的不同 API。 |
| 7107 | `SetSpeedMode(int)` | 0–3 | 运行模式最大速度档位。 |

便捷方法与实际调用的对应关系：

| 便捷方法 | 实际请求 | 说明 |
|---|---|---|
| `ZeroTorque()` | `SetFsmId(0)` | 零力矩。 |
| `Damp()` | `SetFsmId(1)` | 阻尼。 |
| `Squat()` | `SetFsmId(2)` | 位置控制蹲下。 |
| `Sit()` | `SetFsmId(3)` | 位置控制坐下。 |
| `StandUp()` | `SetFsmId(4)` | 源码方法名为 StandUp；官方 FSM 表把 4 描述为 Lock Standing。 |
| `Start()` | `SetFsmId(500)` | 进入 Walk Motion，不是启动 ROS 节点。 |
| `StopMove()` | `SetVelocity(0, 0, 0)` | 发送零速度，默认有效 1 秒。 |
| `BalanceStand()` | `SetBalanceMode(0)` | 零速稳定后停止主动踏步，只在保持平衡需要时迈步。 |
| `ContinuousGait(flag)` | `SetBalanceMode(flag ? 1 : 0)` | 控制机器人是否持续踏步。 |
| `HighStand()`/`LowStand()` | `SetStandHeight()` 的最大/最小无符号哨兵值 | 由固件解释为最高/最低站姿，不是实际米制高度。 |
| `Move(vx, vy, vyaw)` | `SetVelocity(...)` | 默认目标只保持 1 秒。 |
| `SwitchMoveMode(true)` | 只修改客户端本地布尔值 | 不发送 RPC；只让后续 `Move()` 把 duration 从 1 秒改为 864000 秒。 |
| `WaveHand()`/`ShakeHand()` | `SetTaskId(...)` | 触发仓库已知的内置任务；完整列表仍以固件查询结果为准。 |

`ContinuousGait()` 和 `SwitchMoveMode()` 很容易混淆：前者通过 API 7102 改变机器人是否持续踏步，后者只是修改当前 `LocoClient` 对象后续速度命令的有效时长。对手柄或导航控制，建议关闭长时模式，按 5–10 Hz 刷新短 `duration` 指令，并在输入超时后显式调用 `StopMove()`。

### 高层运动模式

#### FSM ID

以下名称来自当前源码封装和与其对应的官方 G1 高层接口。不同固件可能增删模式，切换结果必须用 `GetFsmId()` 或匹配版本的 `/sportmodestate` 确认。

| FSM ID | 官方含义 | 当前源码入口 |
|---:|---|---|
| 0 | Zero Torque，无平衡控制 | `ZeroTorque()` |
| 1 | Damping，无平衡控制 | `Damp()` |
| 2 | Position Control Squat，无平衡控制 | `Squat()` |
| 3 | Position Control Sit Down，无平衡控制 | `Sit()` |
| 4 | Lock Standing，无平衡控制 | `StandUp()` |
| 500 | Walk Motion | `Start()` 或 `SetFsmId(500)` |
| 501 | Walk Motion，3-DoF waist | 仅 `SetFsmId(501)` |
| 702 | Lie Down / Stand Up | 仅通用 `SetFsmId(702)`；无便捷方法 |
| 706 | Balance Squat / Squat Stand | 仅通用 `SetFsmId(706)`；无便捷方法 |
| 801 | Run | 仅通用 `SetFsmId(801)`；官方注明部分 29-DoF 新固件把对应 `ai_sport` 更新为 802，必须以实机版本为准 |

固件会拒绝不安全的跳转。`fsm_mode=0` 表示静态状态，通常允许切换；`fsm_mode=1` 表示动态状态，多数模式切换会被禁止，但阻尼是安全回退。当前 `g1_arm_action_client.hpp` 还特别规定：Arm Action 只支持 FSM ID `{500, 501, 801}`，且 ID 801 下只支持 `fsm_mode` `{0, 3}`；这里的值 3 属于该动作服务的固件约束，不能从通用的 0/1 静动态定义推导其完整语义。

#### Balance Mode 与 Speed Mode

| 模式 | 值 | 含义 |
|---|---:|---|
| `balance_mode` | 0 | Balance Stand：零速稳定后不持续原地踏步。 |
| `balance_mode` | 1 | Continuous Gait：即使速度为零也保持踏步。 |
| `speed_mode` | 0 | Run 模式最大速度 1.0 m/s。 |
| `speed_mode` | 1 | Run 模式最大速度 2.0 m/s。 |
| `speed_mode` | 2 | Run 模式最大速度 2.7 m/s。 |
| `speed_mode` | 3 | Run 模式最大速度 3.0 m/s。 |

#### MotionSwitcher 模式

MotionSwitcher 管理“哪个运动服务占有机器人”，与 `fsm_id` 不是同一层状态机：

| API ID | 方法 | 作用 |
|---:|---|---|
| 1001 | `CheckMode(form, name)` | 查询当前运动服务；空 `name` 表示没有服务占用。 |
| 1002 | `SelectMode(name_or_alias)` | 选择固件已安装的运动服务。 |
| 1003 | `ReleaseMode()` | 释放当前运动服务，供低层控制接管。 |
| 1004 | `SetSilent(bool)` | 设置 MotionSwitcher silent 状态。 |
| 1005 | `GetSilent(bool&)` | 查询 silent 状态。 |

G1 示例能识别 `normal`、`ai`、`advanced` 等名称，并分别显示为 `sport_mode`、`ai_sport`、`advanced_sport`；这只是固件返回名称的映射，不保证每台机器人都安装全部服务。全身低层控制结束后应重新 `SelectMode()` 恢复此前服务，否则 `/api/sport/*` 不会重新可用。

### 低层模式和电机命令

| 字段/机制 | 含义与约束 |
|---|---|
| `LowCmd.mode_pr = 0` | PR 串联模式，脚踝索引解释为 Pitch/Roll；本工作区的 URDF 和 ros2_control 默认要求该模式。 |
| `LowCmd.mode_pr = 1` | AB 并联模式，相同物理索引解释为 A/B 电机；不能把 A/B 角直接当作 URDF Pitch/Roll 角。 |
| `mode_machine` | 机器人硬件模式标识，不是用户随意选择的步态模式。低层示例从最新 `LowState.mode_machine` 读取并回填。 |
| `MotorCmd.mode` | 示例中 1 为使能、0 为禁用；实际可用值应与对应电机/固件协议一致。 |
| `q/dq/tau/kp/kd` | 位置、速度、前馈力矩、位置刚度和速度阻尼目标。低层控制器必须自行保证连续性、限位和稳定性。 |
| CRC | `/lowcmd` 必须按 HG `LowCmd` 布局计算 CRC；使用仓库已有 `motor_crc_hg` 实现，不要对序列化后的任意字节流自行猜测。 |
| 29/35 槽位 | 消息有 35 槽，G1 本体使用实际配置对应的前 29 槽；23-DoF 机型以及锁腰配置还会有不可用索引。 |
| `/arm_sdk` 权重 | `motor_cmd[29].q=0` 交还上肢，`=1` 完全接管；应在静止状态渐变权重，避免目标突跳。该特殊语义不适用于 `/lowcmd`。 |

全身低层控制的一般顺序是：读取稳定的 `/lowstate` 和 `mode_machine`，通过 MotionSwitcher 释放内置运动服务，确认旧 `/lowcmd` 发布者静默，从当前反馈姿态建立首帧目标，再以固定周期发送带 CRC 的命令。退出时先回到稳定站姿并停止低层流，再恢复原 MotionSwitcher 服务。当前 `unitree_g1_ros2_control/G1TopicSystem` 已实现这套接管、静默等待、反馈时效检查和失败回滚，不应在旁边再运行官方低层示例。

### Arm Action、Arm SDK 与示例入口

Arm Action 的 `/api/arm/*` 支持：

| API ID | 方法 | 含义 |
|---:|---|---|
| 7106 | `ExecuteAction(action_id)` | 执行固件预定义动作。 |
| 7107 | `GetActionList(string&)` | 获取当前固件支持的动作列表。 |

若 `/arm_sdk` 正在发布，Arm Action 返回 7400；FSM 不满足条件时返回 7404。旧版错误字符串可能把 `rt/arm_sdk` 写成 `rt/armsdk`，实际 ROS 2 示例话题仍是 `/arm_sdk`。`g1_arm_sdk_dds_example` 通过编译期 `ARM_TYPE` 在 5-DoF 臂和 7-DoF 臂映射间选择，必须与实机硬件一致。

`unitree_ros2_example` 中 G1 专用可执行文件如下：

| 可执行文件 | 控制路径 | 用途/注意事项 |
|---|---|---|
